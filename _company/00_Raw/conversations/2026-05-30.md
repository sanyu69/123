## 📊 $R_{Stability}$ 기반 수익화 목표 업데이트

### 1. 핵심 성과 지표 (KPIs) 설계
수익화 모델의 성공은 '안정성'과 '수익'의 통합 측정에 달려 있습니다. 단순 매출뿐만 아니라, 사용자 경험(UX) 안정성을 매출 기여도로 변환합니다.

| KPI 카테고리 | 핵심 지표 (Metric) | 목표 설정 (Target) | $R_{Stability}$ 연관성 |
| :--- | :--- | :--- | :--- |
| **수익화 (Monetization)** | Pro 티어 전환율 (Conversion Rate) | 5% (MVP 단계), 10% (최종 목표) | 안정적인 경험이 구매 결정에 긍정적 영향을 미친다고 가정. |
| **시스템 안정성 (Stability)** | 평균 HTTP 상태 코드 (Error Rate) | 99.9% 성공률 이상 | 시스템 오류 발생 시 수익화 흐름 차단 방지. |
| **사용자 참여 (Engagement)** | 일일 활성 사용자 (DAU) 대비 IAP 구매율 | DAU당 1.5% 이상 | 안정적인 환경에서 사용자가 지속적으로 게임을 플레이하도록 유도. |
| **ROI** | 고객 생애 가치 (LTV) / 고객 획득 비용 (CAC) | LTV:CAC > 3:1 | 시스템 안정성이 높은 신뢰도를 통해 LTV를 극대화. |

### 2. 단계별 마일스톤 및 실행 계획 (Roadmap & Execution Plan)

#### Phase 1: 안정성 기반 MVP 개발 (Foundation & Stability Lock)
**목표:** $R_{Stability}$-Compliant 디자인 명세와 핵심 API 연동 기능을 구현하여 시스템 안정성을 확보합니다.
*   **주요 작업:**
    1.  **환경 설정 완료:** Next.js 환경 구축 및 필수 패키지 설치 확인. (개발 착수 전 선행 점검)
    2.  **API 안정화:** `api_design_stability_iap.md` 기반의 IAP API 호출 로직 구현 및 HTTP 에러(401 등) 처리 로직을 **최우선으로** 구현합니다. (가장 위험 요소 해결)
    3.  **UI/UX 통합:** `designer.md`에 명시된 $R_{Stability}$ 컬러 팔레트와 Glow 효과를 적용하여 핵심 페이지의 디자인 컴포넌트를 완성합니다.

*   **산출물 연결:** `developer.md`, `designer.md`의 기술 사양을 실제 코드로 변환하는 단계.
*   **측정 지표:** API 호출 성공률 (목표: 100%), UI 로딩 시간 (목표: 2초 이내).

#### Phase 2: IAP 기능 구현 및 검증 (Monetization Implementation)
**목표:** 하이브리드 IAP 패키지 구조(Basic/Pro/VIP)를 실제 결제 시스템과 연결하고, 안정성을 검증합니다.
*   **주요 작업:**
    1.  **패키지 구조 구현:** `business.md`에서 정의한 IAP Tier별 기능 잠금 및 접근 권한 로직을 백엔드와 프론트엔드에 구현합니다.

    3.  **내부 안정성 검증:** 모든 IAP 구매 및 티어 전환 시 발생하는 시스템 오류(HTTP 5xx) 발생 여부를 집중적으로 모니터링하고 기록합니다.

*   **산출물 연결:** `business.md`의 수익화 전략과 MVP 기능 요구사항을 코드로 변환하는 단계.
*   **측정 지표:** IAP 구매 성공률, 시스템 오류율 (Critical Error Rate).

#### Phase 3: 베타 테스트 및 최적화 (Validation & Iteration)
**목표:** 실제 사용자 환경에서 KPI를 측정하고 $R_{Stability}$와 매출 간의 상관관계를 검증합니다.
*   **주요 작업:**

    2.  **데이터 시각화:** 핵심 KPI 대시보드를 구축하여 시스템 안정성 지표와 수익 데이터를 실시간으로 연결하여 모니터링합니다.
    3.  **피드백 루프:** 사용자 피드백을 통해 $R_{Stability}$ 관련 UI 요소의 인게이지먼트 변화를 분석하고 디자인을 미세 조정합니다.

*   **산출물 연결:** `business.md`의 KPI 목표 달성 여부를 검증하는 단계.
*   **측정 지표:** Pro 전환율, DAU당 IAP 구매율, LTV/CAC 비율.

#### Phase 4: 출시 및 확장 (Launch & Scale)
**목표:** 설정된 KPI를 달성하고 지속 가능한 수익화 사이클을 구축합니다.
*   **주요 작업:**

**핵심 원칙:** 모든 티어는 **$R_{Stability}$**에 대한 사용자 인식을 즉각적으로 반영해야 하며, **GLOW & Dynamic Effects**를 통해 프리미엄 경험을 극대화합니다.

## 1. 전략 A: 안정성 기반 티어 (Foundation)
*   **목표:** 최소한의 접근성과 기본적인 시스템 안정성 제공. 신뢰 구축에 중점.
*   **시각적 컨셉:** **클린 & 스테이블 (Clean & Stable)**
    *   **컬러 강조:** 기본 브랜드 컬러(예: 딥 블루 계열)를 주조색으로 사용하며, $R_{Stability}$ 지표는 차분하고 명확하게 표시.
    *   **Dynamic Effects:** 최소화. 인터페이스의 반응성(Responsiveness)과 데이터 로딩 상태에만 미묘한 부드러운 애니메이션(Soft Transition) 적용.
    *   **UI/UX 흐름:** 직관적이고 단순함. 복잡한 설정 옵션은 숨기고, 핵심 기능 접근에 집중.
    *   **핵심 요소:** 안정성 표시(Stability Indicator)가 항상 명확하게 보임.

## 2. 전략 B: 프리미엄 경험 티어 (Experience) - **권고안**
*   **목표:** 시스템 안정성과 고도화된 사용자 경험(UX)을 결합하여 프리미엄 가치 제공 및 수익 극대화.
*   **시각적 컨셉:** **네온 & 다이내믹 (Neon & Dynamic)**
    *   **컬러 강조:** 브랜드 컬러와 네온 계열의 보조 색상(예: 사이버 핑크, 시안)을 활용하여 역동성을 부여. $R_{Stability}$ 지표는 실시간으로 변동하며 '고급 안정성'을 상징적으로 표현.
    *   **Dynamic Effects:** 적극 사용. 마우스 오버 시 GLOW 효과, 데이터 로딩 중에는 부드러운 파티클 애니메이션 적용하여 몰입감 증대.
    *   **UI/UX 흐름:** 기능 잠금 해제 시의 경험이 극적으로 달라지도록 설계. 고급 설정(예: 커스텀 테마, 고급 API 접근)을 매력적인 인터페이스로 노출.
    *   **핵심 요소:** 모든 상호작용에서 '반응성'과 '미학'이 강조됨.

## 3. 전략 C: 엔터프라이즈/완전 제어 티어 (Control)
*   **목표:** 시스템에 대한 완전한 통제권 및 최고 수준의 API 안정성 보증 제공. B2B 잠재력을 고려한 최고 가치 설정.
*   **시각적 컨셉:** **오버로드 & 마스터리 (Overload & Mastery)**
    *   **컬러 강조:** 짙은 배경색(Dark Mode 선호)과 강렬한 대비 색상(예: 블랙/다크 그레이 바탕에 밝은 네온 라인). 이는 시스템의 복잡성과 깊이를 시각적으로 표현.
    *   **Dynamic Effects:** 정밀하고 미세한 제어. 데이터 흐름, 오류 매핑 규칙 등 기술적 정보는 깔끔하게 정리하되, '통제'를 상징하는 미세한 그리드 애니메이션을 사용.
    *   **UI/UX 흐름:** 관리자 대시보드처럼 기능적이고 구조화됨. 모든 $R_{Stability}$ 설정 및 오류 매핑 규칙을 시각적으로 탐색 가능하도록 설계.
    *   **핵심 요소:** 시스템의 깊이와 제어 권한을 암시하는 미니멀하지만 강력한 디자인 언어 사용.

### ⚙️ 기능 잠금(Feature Gating) 구현 계획 초안

**목표:** IAP 티어별 접근 권한을 서버 측에서 엄격하게 관리하고, 비인가된 기능 호출을 차단하여 시스템 안정성($R_{Stability}$)을 확보한다.

#### 1. 핵심 제약 조건 목록 (Constraints for $R_{Stability}$)

이 제약 조건들은 모든 기능 잠금 로직의 기반이 됩니다.

*   **C1: 권한 기반 접근 (Permission-Based Access):** 모든 민감한 기능(예: 프리미엄 코스메틱 구매, 무제한 저장)은 사용자 세션 또는 계정 데이터베이스에서 명시적으로 승인된 티어에 의해서만 호출 가능해야 한다.
*   **C2: 비동기 안정성 (Asynchronous Stability):** IAP 거래 및 권한 변경 요청 시, 데이터베이스 트랜잭션이 완벽하게 성공하거나 실패(Rollback)하는지 확인하고, 그 결과를 사용자에게 즉시 피드백해야 한다. 불안정한 상태에서의 기능 접근은 엄격히 금지한다.
*   **C3: 데이터 무결성 보존 (Data Integrity Preservation):** 잠금 로직은 클라이언트 측에서만 실행되어서는 안 되며, 항상 서버 측의 권한 검증 레이어(Authorization Layer)를 거쳐야 한다. DB에 기록된 상태와 실제 기능 접근 권한이 일치해야 한다.
*   **C4: 에러 매핑 (Error Mapping):** 권한 오류 발생 시, 일반적인 HTTP 403 Forbidden 외에 비즈니스 로직 오류(예: '권한 부족' 대신 '현재 구독 상태 확인 필요')를 명확히 구분하여 사용자에게 전달해야 한다. 이는 $R_{Stability}$의 일관성을 위해 필수적이다.

#### 2. 기능 잠금 로직 설계 (Logic Flow)

**A. 데이터 모델 정의:**
`User` 테이블 또는 `Subscription` 엔티티에 다음 필드를 추가하거나 확장합니다.
*   `user_id`: 사용자 식별자
*   `subscription_tier`: 현재 구독 등급 (Basic, Pro, VIP)
*   `access_status`: 기능 활성화 상태 (예: `pro_features_unlocked`, `vip_content_enabled`)

**B. 권한 검증 함수 설계 (`check_feature_access(user_id, feature_name)`):**
이 함수는 모든 민감한 API 엔드포인트 앞에 위치해야 합니다.

1.  **입력:** 사용자 ID와 요청된 기능 이름.
2.  **검증 단계 1 (세션 확인):** 현재 세션의 유효성과 인증 상태를 검사한다. (기본 보안)
3.  **검증 단계 2 (권한 확인):** 데이터베이스에서 `user_id`에 연결된 `subscription_tier`를 조회한다.
4.  **검증 단계 3 (정책 매핑):** 요청된 `feature_name`과 현재 `subscription_tier` 간의 매핑 규칙(예: 'Ad-free'는 Pro 이상)을 확인한다.
5.  **결과 반환:** 권한이 있으면 `True`, 없으면 `False`를 반환한다.

**C. 트랜잭션 통합 (Transaction Integration):**
IAP 결제 후 기능 잠금 상태를 업데이트하는 과정은 반드시 **원자성(Atomicity)**을 보장해야 합니다.

1.  결제 성공 $\rightarrow$ DB에 구독 정보 업데이트 (트랜잭션 시작)
2.  업데이트된 정보를 기반으로 해당 사용자의 기능 잠금 상태를 즉시 반영한다.
3.  모든 단계가 성공하거나 실패(Rollback)하도록 보장하여 $R_{Stability}$를 유지한다.

 * Feature Gating Service
 * $R_{Stability}$ 기반 접근 제어 로직 구현

interface SubscriptionTier {
    tier: 'Basic' | 'Pro' | 'VIP';
}

// 가격 구조 매핑 (실제 DB 정책과 동기화 필요)
const TIER_MAPPING: Record<SubscriptionTier['tier'], { [key: string]: boolean }> = {
    Basic: {
        'Ad-free': false, // Pro 이상에서 true
        'Unlimited Saves': false, // Pro 이상에서 true
        'Exclusive Content': false // VIP 이상에서 true
    },
    Pro: {
        'Ad-free': true,
        'Unlimited Saves': true,
        'Exclusive Content': false // VIP 이상에서 true
    },
    VIP: {
        'Ad-free': true,
        'Unlimited Saves': true,
        'Exclusive Content': true
    }
};

/**
 * 사용자 권한을 기반으로 기능 접근을 검증합니다.
 * @param userId DB에서 조회된 사용자 ID
 * @param featureName 요청된 기능 이름 (예: 'ad_free', 'unlimited_saves')
 * @returns 기능 접근 가능 여부 (boolean)
 */
export async function checkFeatureAccess(userId: string, featureName: string): Promise<boolean> {
    // 1. 사용자 정보 조회 (DB 호출 시뮬레이션)
    const user = await db.getUser(userId); // 실제 DB 연결 로직 필요

    if (!user) {
        throw new Error("User not found.");
    }

    const tier = user.subscription_tier;

    // 2. 권한 매핑 확인
    const requiredAccess = TIER_MAPPING[tier];
    
    if (!requiredAccess) {
        // 알 수 없는 티어일 경우, 기본적으로 거부 (안정성 우선)
        console.warn(`Unknown tier for user ${userId}: ${tier}`);
        return false;
    }

    const isAllowed = requiredAccess[featureName];

    if (isAllowed === undefined) {
        // 정의되지 않은 기능에 대한 접근 시도 방지
        throw new Error(`Feature ${featureName} is not defined in the current plan.`);
    }

    return isAllowed;
}

#### 1단계: 설계 및 데이터 매핑 (Design & Data Mapping)

1.  **권한 검증 서비스 구현:** 사용자의 구독 상태를 확인하고, 요청된 리소스($R$)가 해당 사용자에게 허용되는지($A$)를 검사하는 핵심 서비스 함수(`check_feature_access(userId, featureId)`)를 개발합니다.

3.  **예외 처리 강화:** 기능이 잠겨있을 경우(Access Denied), 사용자에게 적절한 HTTP 상태 코드(예: 403 Forbidden)와 명확한 에러 메시지를 반환하도록 예외 처리 로직을 강화하여 $R_{Stability}$를 확보합니다.

3.  **성능 및 안정성 모니터링:** 기능 잠금 로직이 전체 API 응답 시간에 미치는 영향을 모니터링하고, 시스템 부하가 증가했을 때에도 $R_{Stability}$ 지표가 유지되는지 확인합니다.

## 🚀 IAP 기능 잠금(Feature Gating)의 $R_{Stability}$ 연관성 정의

기능 잠금은 단순히 접근을 제한하는 것을 넘어, 시스템 안정성을 **위험 기반 분리(Risk-Based Segmentation)**를 통해 보장합니다. 핵심 원칙은 다음과 같습니다.

**원칙:** 기능 잠금은 **시스템 자원의 사용량, 데이터 민감도, 그리고 잠재적 오류 발생 가능성**에 따라 계층화되어야 합니다.

1.  **위험 최소화 (Risk Minimization):** 가장 단순하고 안정적인 기능(Basic Tier)을 기본 접근으로 설정하여, 시스템의 핵심 운영 흐름이 불안정해질 위험을 원천 차단합니다.
2.  **자원 제어 (Resource Control):** 고성능 리소스나 복잡한 데이터 처리 기능을 상위 티어로 잠금으로써, 사용자가 구독 수준에 맞는 컴퓨팅 자원을 사용하도록 유도하고 시스템의 과부하(Overload)를 방지하여 $R_{Stability}$를 높입니다.
3.  **데이터 무결성 (Data Integrity):** 민감한 데이터 접근이나 복잡한 설정 변경 기능은 상위 티어로 제한하여, 잘못된 입력이나 오류가 발생했을 때 시스템 전체의 데이터 무결성에 미치는 영향을 최소화합니다.

## 🛠️ 개발팀을 위한 최종 제약 조건 (Actionable Constraints)

개발팀은 다음 세 가지 핵심 사항을 기준으로 기능 잠금 로직($check\_feature\_access$ 함수 및 미들웨어)을 구현해야 합니다.

1.  **논리적 우선순위 확정:** 모든 API 요청에 대해, **가장 낮은 티어(Basic)**에서 시작하여 상위 티어로 점진적으로 접근 권한을 부여하는 논리를 최우선으로 설계한다.
2.  **서버 측 제어 의무화 (Mandatory Server-Side Control):** 클라이언트 측 UI/UX에 기반한 기능 잠금은 무효이며, 모든 기능 접근 결정은 **서버의 권한 검증 레이어**를 통해서만 이루어져야 한다. 이는 $R_{Stability}$ 확보의 핵심이다.
3.  **예외 처리 강화:** 권한이 없는 요청(403 Forbidden)에 대한 응답 시, 단순히 거부하는 것을 넘어 **'사용자가 접근할 수 있는 다음 단계'** 또는 **'구독 업그레이드 안내'**를 포함하여 사용자 경험($R_{Stability}$의 사용자 신뢰 요소)을 유지해야 한다.
