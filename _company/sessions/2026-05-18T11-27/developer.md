## ⚙️ 병목 모듈 리팩토링 계획 초안: 결제 흐름 (`payment_processor.ts`)

### 🎯 목표
**$R_{\text{Stability}}$ 향상 및 $\Delta T$ (지연 시간) 최소화.** 특히 비동기 처리 중 발생하는 대기 시간을 정량적으로 줄이고, 오류 발생 시 시스템이 안정적으로 복구되도록 보장합니다.

### 🔍 현황 분석 (가정 기반)
현재 `payment_processor.ts`는 결제 요청 $\rightarrow$ 외부 API 통신 $\rightarrow$ 상태 업데이트의 순차적 또는 중첩된 비동기 흐름을 가지고 있으며, 이 과정에서 네트워크 지연이나 서버 응답 대기 시간이 $\Delta T$를 유발하고 있다고 가정합니다.

### 🛠️ 리팩토링 단계별 계획 (Phase 1: 안정화 집중)

#### **Step 1: 지연 시간 측정 로직 정교화 (Measurement Refinement)**
*   **목표:** 현재의 추적 로직(`monitoring_plan.md`에 정의된 $\Delta T$ 수집 지점)이 실제 병목 지점을 정확히 포착하도록 수정합니다.
*   **실행 내용:**
    1.  각 비동기 단계(API 호출, 데이터베이스 쓰기, 클라이언트 응답 대기)별로 시작 시간과 종료 시간을 더 정밀하게 기록하는 타이머를 삽입합니다.
    2.  특히 외부 결제 API 호출 시 발생하는 **네트워크 I/O 대기 시간**을 별도로 측정하도록 로직을 분리합니다.
*   **예상 결과:** 각 단계의 지연 시간($\Delta T$)이 명확히 분리되어, 병목 원인 분석(Root Cause Analysis)이 용이해집니다.

#### **Step 2: 동기화 및 오류 처리 개선 (Synchronization & Error Handling)**
*   **목표:** 비동기 흐름 내에서 발생할 수 있는 잠재적 경쟁 조건(Race Condition)과 예외 상황에 대한 안정성을 강화합니다.
*   **실행 내용:**
    1.  `Promise.all()` 또는 `async/await` 구조를 재검토하여, 모든 병렬 작업이 성공적으로 완료될 때까지 대기하도록 보장하는 안전 장치를 추가합니다.
    2.  외부 결제 API 호출 실패 시, **롤백(Rollback) 메커니즘**을 명시적으로 구현합니다. (예: 결제 요청 후 상태를 'PENDING_FAILED'로 즉시 변경하고 클라이언트에게 알림).
*   **예상 결과:** 예기치 않은 오류 발생 시 시스템이 일관된 안정 상태를 유지하며 복구될 수 있습니다.

#### **Step 3: 데이터 흐름 분리 및 추상화 (Decoupling & Abstraction)**
*   **목표:** 결제 로직과 비즈니스 로직을 분리하여 SRP(단일 책임 원칙)를 준수하고 유지보수성을 높입니다.
*   **실행 내용:**
    1.  결제 관련 외부 API 통신 및 상태 관리를 별도의 **Service Layer**로 추출합니다. (예: `PaymentGatewayService` 신설).
    2.  데이터베이스와의 상호작용(Repository 패턴)을 분리하여, 결제 로직이 데이터 저장소의 세부 구현에 덜 의존하도록 만듭니다.
*   **예상 결과:** 향후 다른 결제 방식이나 시스템 변경 시 영향 범위를 최소화합니다.
