CEO의 지시를 확인했습니다. `business` 에이전트가 제시한 IAP 구조(Free, Pro, VIP)에 맞춰 Phase 1($R_{Stability} \ge 85\%$) 달성을 위한 API 통합 테스트 프레임워크 구축 계획을 구체화하겠습니다. 이 목표는 시스템 안정성($R_{Stability}$)을 수익화 신뢰도의 선행 조건으로 설정한 회사 정체성과 일치합니다.

테스트 프레임워크는 단순히 기능 테스트를 넘어, **데이터 흐름의 무결성**과 **시스템 안정성 지표의 정확성**을 검증하는 데 초점을 맞춰야 합니다.

### ⚙️ Phase 1 목표: $R_{Stability} \ge 85\%$ 달성을 위한 API 통합 테스트 프레임워크 계획

테스트 프레임워크는 다음 세 가지 핵심 모듈로 구성됩니다: **데이터 흐름 검증**, **안정성 지표 검증**, **결제 연동 검증**입니다.

#### 1. 테스트 환경 및 데이터 준비 (Setup & Mocking)

**목표:** 실제 운영 환경과 유사한 조건에서 안정성 데이터를 시뮬레이션하고, 외부 결제 시스템과의 인터페이스를 안전하게 테스트한다.

*   **환경 설정:**
    *   **Staging/Test Environment 구성:** 실제 프로덕션 데이터가 아닌, $R_{Stability}$ 지표가 85% 이상인 상태를 모방하는 가상의 데이터셋(Mock Data)을 준비합니다. 이 Mock 데이터는 시스템이 정상적으로 데이터를 수집하고 API로 전달하는 **경로**와 **포맷**을 테스트하기 위한 기준점이 됩니다.
    *   **API 인터페이스 정의:** $R_{Stability}$ 측정 지표를 반환하는 엔드포인트(`GET /stability/score`)와 인앱 결제(IAP) 상태를 업데이트하는 엔드포인트(`POST /user/{id}/status`)에 대한 계약(Contract)을 명확히 정의합니다.
*   **Mocking 전략:**
    *   **데이터 흐름 Mock:** 데이터 수집 모듈이 API로부터 데이터를 성공적으로 받아 처리하는지 검증하기 위해, 데이터 소스(예: `stability_monitor.py`의 결과)가 API로 전달되는 과정에 대한 Mocking을 수행합니다.
    *   **결제 시스템 Mock:** 실제 PayPal 연동 대신, 결제 성공/실패 시나리오를 모방하는 Mock 결제 게이트웨이 함수를 구현하여, API 호출이 정상적인 비즈니스 결과(성공 또는 실패)를 반환하는지 검증합니다.

#### 2. 핵심 테스트 모듈 (Core Testing Modules)

**목표:** $R_{Stability}$의 실시간 반영 및 수익화 흐름에 직접 영향을 미치는 지점들의 안정성을 보장한다.

##### A. Stability Data Integrity Test (안정성 데이터 무결성 검증)
*   **테스트 대상:** 시스템이 계산한 $R_{Stability}$ 값과 API를 통해 반환되는 값이 일치하는지 확인합니다.
*   **실행 내용:** Mock 데이터셋을 사용하여, 안정성 모니터링 로직($R_{Stability}$ 계산) $\rightarrow$ 데이터베이스 저장 $\rightarrow$ API 응답 과정을 엔드투엔드로 테스트합니다.
*   **통과 조건:** 모든 단계에서 $R_{Stability}$ 값의 손실이나 오차가 발생하지 않아야 합니다.

##### B. IAP & Stability Tier Logic Test (티어 로직 검증)
*   **테스트 대상:** 사용자 등급(Free/Pro/VIP)에 따라 접근 권한 및 API 응답이 정확하게 분리되는지 확인합니다.
*   **실행 내용:** 특정 사용자가 Pro 상태일 때, $R_{Stability}$ 대시보드 데이터(`GET /stability/dashboard`)에 접근하는지, 광고 제거 기능(Pro 전용)이 활성화되는지 등을 API 호출을 통해 검증합니다.
*   **통과 조건:** 사용자 티어 변경 시, API 응답의 권한 수준 변화가 즉각적이고 정확하게 반영되어야 합니다.

##### C. Transaction Flow Test (결제 흐름 검증)
*   **테스트 대상:** 인앱 결제(IAP) 성공/실패 시나리오가 시스템 안정성에 미치는 영향을 확인합니다.
*   **실행 내용:** Mock된 결제 실패 상황을 주입하여, 시스템이 비정상적인 상태로 전환되지 않고 안전하게 오류를 처리하는지 검증합니다. 특히 $R_{Stability}$ 지표에 부정적인 영향이 발생하지 않도록 격리 테스트가 필수입니다.
*   **통과 조건:** 결제가 실패하더라도 시스템의 안정성($R_{Stability}$) 데이터는 손상되거나 불안정해지지 않아야 합니다 (시스템 안정성 우선 원칙 준수).

#### 3. 자동화 및 반복 루프 구축 (Automation & Loop)

*   **테스트 스크립트 작성:** 위에서 정의한 각 모듈에 대해 `pytest` 또는 유사한 프레임워크를 사용하여 테스트 케이스를 코드로 명시합니다.
*   **CI/CD 통합:** 이 테스트 스크립트를 CI 파이프라인(예: GitHub Actions)에 통합하여, 코드 변경 시 자동으로 해당 API 연동 및 안정성 검증을 실행하도록 설정합니다.
