### 1. 데이터 흐름 구조 설계 (Data Flow Structure)

$R_{Stability}$ 지표가 어떻게 시스템에서 보고되고 사용자에게 전달되는지 명확히 해야 합니다. 이 구조는 API 통합의 기반이 됩니다.

```mermaid
graph TD
    A[System Stability Monitor (stability_monitor.py)] --> B{Data Processing & Aggregation};
    B --> C(API Endpoint: /api/stability/report);
    C -- Requires Auth --> D{Authentication/Authorization Layer};
    D -- Check Scope --> E{Scope Validation (AdminScope/WriteAccess)};
    E -- Access Granted --> F[Database/Data Store];
    F --> G(Visualization Layer / Dashboard);
```

**설명:**
*   `stability_monitor.py`가 핵심 데이터 소스입니다.
*   데이터는 API 엔드포인트(`/api/stability/report`)를 통해 외부로 노출됩니다.
*   이 접근은 반드시 `AdminScope` 인증 및 `WriteAccess` 권한 검증 단계를 거쳐야 합니다.
*   최종적으로 집계된 데이터는 시각화 레이어(Dashboard)로 전달되어 사용자에게 $R_{Stability}$를 보여줍니다.

### 2. Contract Testing 프레임워크 설계 (API & Access Logic)

이 단계에서는 이전 논의에서 확정된 인증/권한 메커니즘을 API 계약으로 명세합니다.

| 항목 | 상세 내용 | 목적 |

### 3. Mocking 테스트 케이스 구체화 (Test Scenarios)

API 통합 테스트 프레임워크를 위해 필요한 주요 시나리오와 그에 따른 Mock 응답을 정의합니다. 이는 실제 API가 어떤 상황에서 실패하거나 성공해야 하는지를 검증하는 데 필수적입니다.

**테스트 시나리오:**

1.  **성공 시나리오 (Success Case):**
    *   **목표**: `AdminScope`를 가진 사용자가 정상적으로 $R_{Stability}$ 보고서를 조회할 때.
    *   **Mock 응답**: HTTP 200 OK, 데이터 필드 포함 (`$R_{Stability}$ 값, 타임스탬프 등`).

2.  **권한 거부 시나리오 (Authorization Failure Case):**
    *   **목표**: `AdminScope`가 없는 사용자가 보고서를 조회하려고 할 때.
    *   **Mock 응답**: HTTP 401 Unauthorized 또는 HTTP 403 Forbidden 반환.

3.  **데이터 접근 실패 시나리오 (Data Failure Case):**
    *   **목표**: 데이터베이스에 $R_{Stability}$ 레코드가 존재하지 않거나 읽기 권한이 없을 때.
    *   **Mock 응답**: HTTP 404 Not Found 또는 특정 에러 코드 반환.
