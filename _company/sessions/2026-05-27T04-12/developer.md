### 🎯 행동 계획: 안정성 우선 MVP 설계 (Stability-First MVP Design)

현재 목표는 '완성'보다 '안정적인 시작'에 초점을 맞춥니다. 데이터 파이프라인의 불안정성을 최소화하면서 웹사이트의 핵심 기능(MVP)을 정의합니다.

#### 1단계: $R_{Stability}$ 확보를 위한 최소 기술 마일스톤 (MVP 정의)
**목표:** 실제 비즈니스 로직 구현보다 시스템 안정성 검증에 집중할 수 있는 최소한의 범위 설정.

| 마일스톤 | 목표 범위 | 핵심 요구사항 ($R_{Stability}$ 기여도) | 의존성 |
| :--- | :--- | :--- | :--- |
| **MVP 1: 프론트엔드 레이아웃 (Static Shell)** | 웹사이트의 시각적 구조와 네비게이션, IAP 버튼 배치를 포함하는 정적 HTML/CSS 구조. | 시각 일관성 및 초기 로딩 성능 검증 (디자이너 작업 기반). | $R_{Stability}$ 확보 필요 없음 (환경 설정만 요구됨). |
| **MVP 2: 핵심 상호작용 로직 (Mock Interaction)** | IAP 결제 흐름의 UI/UX 시뮬레이션. 실제 PayPal 연동 없이, 클라이언트 측에서 상태 변화를 시뮬레이션하는 기능 구현. | 사용자 흐름(Conversion Flow)의 논리적 오류 방지 및 인터페이스 안정성 검증. | $R_{Stability}$ 확보 필요 없음 (로직 분리). |
| **MVP 3: 데이터 수집 스켈레톤 (Data Skeleton)** | 실제 데이터베이스 연결 없이, 시스템이 데이터를 수신할 **API 엔드포인트 정의** 및 **데이터 구조(Schema) 설계**. | API 인증 및 데이터 형식의 안정성 확보. $R_{Stability}$에 직접 기여. | $R_{Stability}$ 확보 필요함 (API 연결 준비). |

#### 2단계: 데이터 수집 로직 최소 요구사항 정의
**목표:** 실제 서비스 시작 전, 데이터 파이프라인이 안정적으로 작동할 최소한의 규칙을 정의합니다.

1.  **데이터 소스 명확화:** 어떤 종류의 데이터($R_{Stability}$ 지표, 사용자 행동 로그 등)가 필요한지 식별하고, 각 데이터 소스가 **어떤 형식**으로 코다리 시스템에 입력되어야 하는지 (JSON Schema 또는 TypeScript Interface로 정의).
2.  **에러 핸들링 명세:** 데이터 수집 과정에서 발생하는 모든 잠재적 오류(API 타임아웃, 인증 실패, 잘못된 형식 등)에 대한 **예외 처리 및 재시도 로직의 최소 사양**을 정의합니다. (이는 $R_{Stability}$ 확보에 직결됨).
3.  **실시간 반영 기준:** 시스템 안정성 지표($R_{Stability}$)가 데이터 파이프라인에 실시간으로 반영되는 과정에서, **어떤 지표가 필수적으로 반영되어야 하는지**의 우선순위를 정의합니다. (예: Latency, Error Rate 등 핵심 지표만 우선).
