### 1. 재검토된 핵심 마일스톤 (Phase 2/3 통합)

가장 시급한 마일스톤은 **"시스템 안정성($R_{Stability}$)이 반영된 데이터 파이프라인과 프론트엔드 UI의 완벽한 동기화 및 비즈니스 로직의 무결성 검증"**으로 설정합니다.

| 마일스톤 | 목표 | $R_{Stability}$ 연관성 | 핵심 산출물 |
| :--- | :--- | :--- | :--- |
| **Phase 2: 데이터 동기화 통합** | 백엔드에서 생성/수정되는 모든 핵심 데이터($R_{Stability}$ 지표 포함)가 프론트엔드로 정확하게 표시되고 상호작용이 이루어지도록 구현. | 데이터 무결성 및 실시간성 보장. (데이터 불일치 방지) | API 응답에 대한 E2E 검증 스위트 |
| **Phase 3: 비즈니스 로직 통합** | 인앱 결제(PayPal)와 같은 핵심 비즈니스 흐름이 시스템 안정성 제약 조건 하에서 오류 없이 실행되는지 최종 검증. | 트랜잭션의 원자성 및 오류 처리 안정성 확보. (수익화 모델의 신뢰도 확보) | 트랜잭션 기반 E2E 시나리오 |

### 2. 구체적인 구현 계획 (Phase 2 집중)

Phase 2인 **데이터 동기화 통합**에 중점을 두고, 다음의 기술적 흐름을 제안합니다.

1.  **API Contract 정의 및 검증:**
    *   백엔드 API 엔드포인트($R_{Stability}$ 지표 반환 포함)와 프론트엔드가 기대하는 데이터 구조(Schema)를 명확히 정의하고, 이를 기반으로 자동화된 계약 검증 로직을 설계합니다.
2.  **데이터 흐름 모듈 구현:**
    *   $R_{Stability}$ 데이터를 서버에서 받아와 클라이언트가 요청할 때 실시간 또는 주기적으로 업데이트되는 데이터 파이프라인을 확립합니다. (WebSocket 또는 Polling 전략 고려)
3.  **상태 관리 동기화 로직:**
    *   클라이언트 상태(State)와 서버의 안정성 지표($R_{Stability}$) 간의 일관성을 유지하기 위한 클라이언트 측 검증 로직을 설계하고 구현합니다.

### 3. 초기 테스트 스위트 설계 (E2E & 통합 중심)

$R_{Stability}$ 목표 달성을 위해, 단순 기능 테스트를 넘어 **'안정성 기반 시나리오'**를 포함하는 테스트 스위트를 설계합니다.

#### A. 데이터 무결성 검증 스위트 ($R_{Stability}$ Focus)

*   **테스트 케이스 1: 정상 흐름 검증 (Happy Path)**
    *   시나리오: 시스템이 정상적으로 작동하며, $R_{Stability}$ 지표가 예상대로 업데이트되고 UI에 정확히 반영되는지 확인.
    *   검증 항목: 데이터 수신 성공 여부, Latency 측정 (최대 허용치 준수), UI 렌더링 정확성.
*   **테스트 케이스 2: 경계값 및 스트레스 검증 (Boundary & Stress)**
    *   시나리오: $R_{Stability}$ 지표가 최대/최소값에 도달했을 때 시스템이 안정적으로 응답하는지 확인합니다. API에 과도한 데이터 부하를 주입하여 시스템의 반응 속도와 오류 처리 방식을 검증합니다.
    *   검증 항목: 극단적인 값 입력 시 에러 발생 여부, 응답 시간 변화율.
*   **테스트 케이스 3: 오류 주입 및 복구 검증 (Failure Injection & Recovery)**
    *   시나리오: 데이터 수집 모듈(가상)에 의도적으로 오류를 주입하거나(예: API 연결 끊김, 잘못된 데이터 포맷 반환), 서버 측에서 임의의 오류 코드(예: 500 에러)를 발생시킵니다.
    *   검증 항목: 프론트엔드가 오류 상태를 안정적으로 처리하고 사용자에게 명확한 메시지를 제공하는지 확인합니다. (Graceful Degradation 테스트 포함)

#### B. 비즈니스 로직 통합 검증 스위트 (Payment Flow Focus)

*   **테스트 케이스 4: 인앱 결제 무결성 검증 (Transaction Integrity)**
    *   시나리오: 사용자가 Basic $\to$ Pro로 전환을 시도할 때, 서버의 트랜잭션 로직(결제 성공/실패 여부)과 클라이언트 측 상태 업데이트가 완벽하게 일치하는지 확인합니다.
    *   검증 항목: 결제 성공 후 사용자 티어 변경, 데이터베이스 레코드 업데이트, 세션 관리의 동기화.
*   **테스트 케이스 5: 안정성 기반 접근 제어 검증 (Stability-based Access)**
    *   시나리오: $R_{Stability}$ 지표가 특정 임계값(예: Pro 티어 요구치)을 충족하지 못할 때, 결제 또는 기능 접근이 차단되는지 확인합니다.
    *   검증 항목: 안정성 지표에 따른 권한 부여 로직의 정확성.
