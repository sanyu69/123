### 1. $R_{Stability}$ 기반의 구현 요구사항 검토 (전략적 병목 식별)

시스템 안정성($R_{Stability}$)을 최우선 가치로 설정했으므로, PayPal 연동 흐름에서 발생할 수 있는 가장 큰 위험 요소는 **결제 성공/실패 시점에서의 상태 관리**와 **재시도 로직의 부재**입니다.

| 검토 항목 | 요구사항 (`payment_flow_design.md` 기반 가정) | $R_{Stability}$ 영향 | 잠재적 병목 지점 |
| :--- | :--- | :--- | :--- |
| **트랜잭션 상태 관리** | 결제 요청 → PayPal 응답 수신 → 서버 DB 업데이트 (성공/실패 플래그 설정) | 높음 | 비동기 처리 중 상태 불일치(Race Condition). 결제가 성공했으나 DB에 반영되지 않거나, 실패했으나 재시도 로직이 작동하지 않는 경우. |
| **에러 핸들링** | PayPal API 호출 실패 (인증 오류, 한도 초과 등) 시의 명확한 에러 코드 정의 및 사용자 피드백 제공 | 매우 높음 | 예외 처리 누락으로 서버가 멈추거나 잘못된 상태로 전환될 위험. 특히 `exit 9009`와 같은 시스템 레벨 에러 발생 시 복구 메커니즘 부재. |
| **보안 및 인증** | API 키/토큰의 안전한 관리 및 민감 데이터(결제 정보)의 최소화된 처리 | 높음 | 환경 변수 또는 파일 내 하드코딩으로 인한 보안 취약점. (PAYPAL_CLIENT_ID, SECRET 관리) |
| **재시도 로직** | 일시적인 네트워크 문제나 PayPal 서버 지연에 대한 자동 재시도 메커니즘 설계 여부 | 중간 | 사용자는 결제 실패를 경험하지만 시스템은 멈추는 상황 발생. 안정성을 저해하는 직접적 원인. |

### 2. 최종 검토 결과 및 권고 (Actionable Insight)

개발팀이 코딩을 시작하기 전에 반드시 다음 두 가지 사항을 **최우선 구현 요구사항**으로 명시하도록 지시해야 합니다.

**A. 결제 흐름의 Atomic(원자적) 보장:**
PayPal API 호출과 데이터베이스 트랜잭션 업데이트는 **단일 원자적 작업 단위(Atomic Unit)**로 묶여야 합니다. 즉, 결제가 성공적으로 완료되기 전까지 서버 상태가 중간 단계에 머무르는 것을 방지해야 합니다.

*   **요구사항:** 모든 결제 흐름은 트랜잭션 시작 시점부터 최종 상태 업데이트까지 **Rollback(취소)** 기능이 명시되어야 합니다. 실패 발생 시, 시스템은 항상 초기 안전 상태로 복원되어야 합니다. (이는 $R_{Stability}$의 핵심 지표입니다.)

**B. 예외 기반의 안정화 로직 강제:**
단순히 에러 메시지를 출력하는 것을 넘어, 모든 외부 서비스(PayPal) 통신 실패 시 **시스템이 스스로 복구 또는 재시도**하도록 설계해야 합니다.

*   **요구사항:** PayPal API 호출에 대해 최소 3회의 지능적인 재시도 메커니즘을 구현하고, 이 과정에서 발생하는 모든 에러(HTTP 상태 코드, 인증 오류 등)는 명확하게 분류되어 로그로 기록되어야 합니다. 실패 시 사용자에게는 친절한 메시지를 제공하되, 서버 내부에서는 **시스템이 멈추지 않도록** 하는 것이 우선입니다.

### 3. 잠재적 병목 지점 요약

1.  **비동기/상태 관리의 불일치:** 결제 성공 여부와 데이터베이스 상태 간의 동기화 실패가 가장 큰 시스템 안정성 저해 요인입니다.
2.  **외부 API 의존성 위험:** PayPal 서버의 일시적인 문제나 인증 오류에 대한 강력한 예외 처리 및 재시도 로직이 없으면, 사용자는 결제를 시도할 때마다 시스템 불안정(멈춤)을 경험하게 됩니다.
