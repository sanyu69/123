💻 코다리: 작업 시작합니다.

CEO의 지시 사항을 분석했습니다. 핵심은 시스템 안정성($R_{Stability}$)을 기반으로 Latency와 Error Rate에 대한 구체적인 측정 기준($C_{min}$)을 정의하고, 가장 위험한 3가지 병목 지점에 대해 최소 실행 가능 안정성 목표치 및 코드 요구사항을 산출하는 것입니다.

이는 단순한 수치 제시가 아니라, 기술적 제약 조건이 곧 수익화 가치로 연결된다는 우리의 핵심 원칙($R_{Stability}$ 기반 가격 책정)에 직접적으로 기여하는 작업입니다.

기존 세션에서 확보된 지식과 목표를 바탕으로 즉시 분석 결과를 산출하겠습니다.

### 1. Latency 및 Error Rate에 대한 $C_{min}$ 측정 기준 정의

$C_{min}$은 시스템 안정성($R_{Stability}$)을 보장하기 위한 기술적 최소 요구 사항입니다. 이는 **사용자 경험(UX) 부정적 영향**과 **최소 투자 비용**을 기준으로 설정되어야 합니다.

| 지표 | $C_{min}$ 목표치 (기준) | $R_{Stability}$ 연관성 | 측정 기준 (Latency/Error Rate) |
| :--- | :--- | :--- | :--- |
| **Latency (응답 시간)** | **P95 < 200ms** | 사용자 이탈률(Churn Rate) 최소화 | API 요청부터 최종 UI 렌더링 완료까지의 95% 응답 시간. |
| **Error Rate (오류율)** | **< 0.1%** | 데이터 신뢰도 및 수익화 근거 확보 | 전체 요청 중 오류 발생 건수 비율. (HTTP 4xx/5xx 에러 포함) |
| **시스템 안정성($R_{Stability}$)** | **99.9% 이상 가용성** | 프리미엄 가격 책정의 기술적 근거 | 서비스가 정상적으로 작동하는 시간의 비율. |

*   **설명:** $C_{min}$은 단순한 기술 스펙이 아니라, 시장에서 요구되는 '프리미엄 경험'을 제공하기 위한 최소 허용치입니다. 200ms 이하의 Latency는 사용자가 지루함을 느끼기 전에 다음 단계로 넘어갈 수 있게 하며, 0.1% 미만의 Error Rate는 데이터 무결성을 보장합니다.

### 2. 가장 높은 위험도를 가진 3가지 병목 지점 및 안정성 목표 산출

최근의 개발 흐름과 $R_{Stability}$에 대한 강조를 고려할 때, 가장 높은 위험도는 **데이터 파이프라인의 안정성**, **API 인증/접근 권한 처리**, 그리고 **클라이언트-서버 간의 동기화**입니다.

| 순위 | 병목 지점 (Bottleneck) | 위험도 평가 | 최소 실행 가능 안정성 목표치 ($C_{min}$) | 최소 코드 스니펫 요구사항 |
| :--- | :--- | :--- | :--- | :--- |
| **1** | **데이터 파이프라인 접근 및 인증** | **최고 (수익화 직결)** | Error Rate: < 0.05% (인증 실패) | API Gateway 또는 백엔드에서 토큰 검증 로직의 불변성 확보 및 재시도(Retry) 메커니즘 구현. |
| **2** | **실시간 데이터 스트리밍 Latency** | 높음 (UX 직결) | Latency: P95 < 150ms | 비동기 데이터 수집 모듈에서 지연 발생 시 타임아웃 및 버퍼링 로직을 명시적으로 정의. |
| **3** | **클라이언트-서버 상태 동기화** | 중간 (내부 오류 방지) | Error Rate: < 0.1% (상태 불일치) | 클라이언트 측에서 서버 상태 업데이트 후, 비동기적으로 상태 일관성 검증(Consistency Check)을 수행하는 로직 분리. |

### 3. 최소 코드 스니펫 요구사항 상세화

**병목 지점 1: 데이터 파이프라인 접근 및 인증 안정화**

*   **목표:** HTTP 401 오류 발생 시 시스템이 자체적으로 재시도하고, 실패 시 명확한 에러 코드를 반환하여 클라이언트의 오해를 방지해야 합니다.
*   **요구사항:** 백엔드 API 핸들러에 **멱등성(Idempotency)**을 보장하는 트랜잭션 레이어를 적용하고, 인증 실패 시 적절한 HTTP 상태 코드와 함께 잠금(Lock) 메커니즘이 작동하도록 구현해야 합니다.
*   **필수 스니펫 방향:** `auth_middleware.py` 또는 해당 API 함수 내부에 **재시도 횟수 제한 및 트랜잭션 롤백/커밋 로직**을 명확히 삽입해야 합니다.

**병목 지점 2: 실시간 데이터 스트리밍 Latency 최소화**

*   **목표:** 데이터 수집 과정에서 발생하는 병목이 최종 사용자 응답 시간을 침해하지 않도록, 데이터 흐름에 대한 시간 제약을 명시적으로 설정해야 합니다.
*   **요구사항:** 스트리밍 데이터 처리 로직 내부에 **최대 지연 허용치(Timeout)**를 설정하고, 이 시간이 초과될 경우 즉각적으로 경고 메시지를 발생시키거나 안정적인 이전 상태를 반환하도록 구현해야 합니다.
*   **필수 스니펫 방향:** `stream_processor.py`의 데이터 수신 루프에 **명시적인 `timeout` 파라미터를 적용**하고, 타임아웃 발생 시 예외 처리 블록을 명확히 분리해야 합니다.

**병목 지점 3: 클라이언트-서버 상태 동기화 오류 방지**

*   **목표:** 클라이언트와 서버 간의 상태 불일치로 인한 내부적 오류를 원천 차단합니다.
*   **요구사항:** 클라이언트가 상태 변경 요청을 보낼 때, 서버는 해당 변경이 유효한지 **상태 검증(State Validation)** 단계를 거쳐야 하며, 동기화 실패 시 즉시 클라이언트에게 명확한 오류 응답을 전달해야 합니다.
*   **필수 스니펫 방향:** 상태 업데이트 함수에 **`validate_state(new_state)`**와 같은 독립적인 검증 함수를 도입하고, 이 검증 로직이 모든 쓰기 작업의 선행 조건이 되도록 강제해야 합니다.
