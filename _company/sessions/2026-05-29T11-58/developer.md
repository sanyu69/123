이 작업은 앞서 수립된 **Code-Ready Spec**과 **기술적 안정성($R_{Stability}$) 기준($C_{min}$)**을 기반으로, 시스템의 안정성을 측정하고 잠재적인 병목 지점(Bottleneck)을 식별하는 데 필요한 구체적인 기술 사양을 정의하는 작업입니다.

저는 다음 세 가지 핵심 요소를 포함하는 모니터링 로직과 최소 요구사항을 정리하여 전달하겠습니다.

### 1. $R_{Stability}$ 기반 실시간 $\Delta T$ 모니터링 로직 정의

시스템 안정성($R_{Stability}$) 지표를 실시간으로 반영하기 위해 측정해야 할 핵심 지표와 그 계산 방식을 정의합니다.

*   **핵심 지표**:
    *   $\Delta T_{\text{latency}}$ (요청 처리 지연 시간): API 응답 또는 데이터 파이프라인 접근에 걸리는 실제 시간을 측정합니다.
    *   $\Delta T_{\text{error}}$ (오류 발생률): 특정 시점 동안 발생한 HTTP 4xx/5xx 에러 또는 파일 시스템 접근 오류의 빈도를 측정합니다.
    *   $R_{Stability} = f(\Delta T_{\text{latency}}, \Delta T_{\text{error}})$: 이 두 지표를 통합하여 시스템 안정성 점수를 산출합니다.

*   **모니터링 로직 (Pseudo-Code)**:
    1.  **데이터 수집**: 각 요청/작업의 시작 시간과 종료 시간을 기록하고, 발생한 예외(Exception) 및 에러 코드를 실시간으로 캡처합니다.
    2.  **지연 시간 계산**: $(\text{End Time} - \text{Start Time}) \rightarrow \Delta T_{\text{latency}}$를 계산합니다.
    3.  **오류율 계산**: 측정 주기($T$) 동안 발생한 총 오류 수($E$)와 전체 요청 수($N$)를 이용하여 $\Delta T_{\text{error}} = E/N$을 계산합니다.
    4.  **안정성 점수 산출**: $R_{Stability} = (\alpha \cdot (1 - \Delta T_{\text{error}})) + ((1-\alpha) \cdot e^{-\beta \cdot \Delta T_{\text{latency}}})$ (여기서 $\alpha, \beta$는 $C_{min}$ 기준에 따라 정의되는 가중치 상수입니다.)

### 2. 최소 코드 스니펫 요구사항 (Minimum Code Snippets)

실제 구현을 위해 필요한 백엔드(Python/FastAPI 또는 Node.js 기반이라고 가정)의 최소한의 인터페이스 및 모니터링 프레임워크 요구사항입니다.

*   **데이터 파이프라인 접근 안정화 로직**:
    *   인증 오류($HTTP\ 401$) 발생 시, 즉시 $R_{Stability}$ 모니터링 프로세스에 경고를 발생시키고 해당 요청을 차단하는 Fail-Fast 메커니즘이 필수입니다. (이는 이전 작업에서 확보된 인증 안정화 로직의 연장선입니다.)
    *   파일 시스템 접근 오류($Permission\ Denied$, $File\ Not\ Found$) 발생 시, 이는 **$R_{Stability}$ 지표에 즉각적인 $-100\%$ 패널티**를 부여하는 트리거로 작동해야 합니다.

*   **API 엔드포인트 요구사항**:
    *   모든 핵심 API 엔드포인트(`/api/v1/...`)는 요청 시작 시 **타임스탬프와 사용자 ID**를 로그 시스템에 기록하도록 강제되어야 하며, 응답 시 **처리 시간($\Delta T_{\text{latency}}$)**을 포함해야 합니다.
    *   모니터링 데이터 수집을 위한 별도의 비동기/백그라운드 작업(Worker)이 API 요청과 독립적으로 실행되어야 합니다.
