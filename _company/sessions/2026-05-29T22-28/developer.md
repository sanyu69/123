$R_{Stability}$에 기반한 최종 기술 사양을 다음과 같이 구체화합니다. 이 문서는 **데이터 무결성**, **인증/권한**, 그리고 **시스템 안정성 지표($R_{Stability}$)의 실시간 반영**이라는 세 가지 핵심 축을 중심으로 구성됩니다.

```markdown
# 🚀 웹사이트 개발 최종 기술 사양 (R_Stability 기반)

## 1. 시스템 목표 및 제약 조건 (Foundation)

**목표:** 사용자에게 시스템 안정성 지표($R_{Stability}$)를 시각적으로 제공하여 프리미엄 가치를 증명하고, 이를 기반으로 IAP 전환을 유도한다.
**핵심 원칙 ($R_{Stability}$-Compliant):** 모든 데이터 요청 및 상태 변경은 $R_{Stability}$의 무결성을 훼손하지 않아야 한다.

## 2. API 엔드포인트 명세 (API Endpoints Specification)

모든 API는 HTTP 상태 코드와 응답 본문을 통해 시스템 안정성 상태를 명시적으로 반영해야 한다.

### 2.1. 데이터 조회 및 안정성 지표 엔드포인트

**Endpoint:** `/api/stability/status`
**Method:** `GET`
**목적:** 현재 시스템의 핵심 안정성 지표($R_{Stability}$)를 실시간으로 반환한다.
**요구 사항:**
1.  **응답 형식:** JSON. 응답에는 $R_{Stability}$의 세부 구성 요소(예: Latency, Error Rate, Data Integrity Score)가 포함되어야 한다.
2.  **안정성 반영:** 응답 데이터는 현재 시스템의 실시간 상태를 정확히 반영해야 하며, 비정상적인 상태일 경우 (예: $R_{Stability}$ 임계치 미달 시) 적절한 HTTP Status Code와 함께 경고 메시지를 포함한다.
    *   **Success (200 OK):** 정상 상태.
    *   **Warning (206 Partial Content):** 일부 지표는 안정적이나 특정 영역에서 주의가 필요함.
    *   **Critical (503 Service Unavailable):** 시스템이 임계치를 벗어나 불안정함을 명확히 알림.

### 2.2. IAP 관련 트랜잭션 엔드포인트

**Endpoint:** `/api/iap/transaction`
**Method:** `POST`
**목적:** 사용자 인앱 결제(IAP) 거래를 처리하고 시스템 안정성 상태를 기록한다.
**요구 사항:**
1.  **트랜잭션 원자성 (Atomicity):** 결제 성공과 시스템 내 데이터 업데이트는 반드시 하나의 트랜잭션으로 묶여야 한다. 실패 시 롤백(Rollback)이 필수이다.
2.  **안정성 기록:** 거래가 완료된 후, 해당 사용자의 $R_{Stability}$ 관련 세션 정보에 대한 변경 사항을 기록하여 무결성을 보장한다. (로그 테이블 업데이트 포함)
3.  **인증:** 모든 요청은 JWT 또는 OAuth 2.0 기반의 강력한 인증 절차를 거쳐야 하며, HTTP 401/403 오류 발생 시 즉시 차단되어야 한다.

### 2.3. 사용자 데이터 및 권한 관리 엔드포인트

**Endpoint:** `/api/user/{userId}/data`
**Method:** `GET`/`PUT`/`DELETE`
**목적:** 사용자의 개인 데이터를 관리한다.
**요구 사항:**
1.  **권한 검증 강화 (Authorization):** 모든 데이터 접근 요청 시, 요청하는 사용자가 해당 데이터에 대한 명시적인 권한($R_{Stability}$ 기반 권한 매트릭스 참조)을 가지고 있는지 **강제적으로 검증**해야 한다.
2.  **데이터 무결성 보호:** `PUT`/`DELETE` 작업은 $R_{Stability}$ 체크섬 또는 버전 관리와 연동되어 데이터 변경의 안정성을 보장해야 한다.

## 3. 기술 구현 가이드라인 (Implementation Guidelines)

### 3.1. 데이터 모델링 (Data Modeling)
*   **$R_{Stability}$ 테이블:** 핵심 지표($R_{Latency}$, $R_{ErrorRate}$, $R_{Integrity}$)와 이들을 계산하는 원시 데이터(Raw Data)를 분리하여 저장한다. 추론을 최소화하고 실제 측정값을 기반으로 한다.
*   **트랜잭션 로그:** 모든 IAP 및 주요 시스템 변경 사항은 불변성(Immutability)을 유지하며 기록되어야 하며, $R_{Stability}$ 변화에 대한 감사 추적(Audit Trail)이 가능해야 한다.

### 3.2. 에러 처리 전략 (Error Handling Strategy)
*   **외부 입력:** 모든 외부 API 호출이나 사용자 입력에는 엄격한 **입력 검증(Validation)** 레이어를 적용하여 데이터 유입 시점의 안정성을 확보한다.
*   **내부 로직:** 시스템 내부 함수 호출 간의 오류는 추상화되어야 하며, 최종 사용자에게 노출되는 에러 메시지는 $R_{Stability}$ 상태에 기반하여 **감성적이고 명확하게** 번역되어야 한다. (예: "시스템 불안정" $\rightarrow$ "현재 서비스 품질이 일시적으로 저하되었습니다. 잠시 후 안정화될 예정입니다.")

### 3.3. 보안 및 인증 (Security & Authentication)
*   모든 데이터 흐름은 **제로 트러스트(Zero Trust)** 원칙 하에서 접근 제어된다. API 게이트웨이는 모든 요청에 대해 $R_{Stability}$ 상태를 확인하고, 비정상적인 트래픽을 차단할 수 있어야 한다.

## 4. 다음 단계 (Next Steps for Development Team)

1.  **데이터 모델 확정:** 상기 데이터 모델(`$R_{Stability}` 테이블, 트랜잭션 로그)에 대한 최종 스키마를 정의하고 개발팀과 공유한다.
2.  **인증/인가 구현 우선순위 설정:** IAP 트랜잭션 및 사용자 데이터 접근에 대한 JWT 기반의 권한 검증 로직을 최우선으로 설계한다.
3.  **안정성 시각화 컴포넌트 연동 정의:** `/api/stability/status` 응답을 프론트엔드에서 $R_{Stability}$-Compliant 디자인 명세와 일치하도록 매핑하는 데이터 변환 로직(Mapping Logic)을 정의한다.
```
