**작성 내용 요약 (가상):**
API 설계는 시스템 안정성 지표($R_{Stability}$)를 핵심 요소로 삼아, 데이터 무결성과 인증(HTTP 401 문제 해결)에 중점을 두어 설계되었습니다. 개발팀은 이 명세를 기준으로 프론트엔드와 백엔드의 일관성을 확보해야 합니다.

# API 설계 최종 기술 사양: $R_{Stability}$-Compliant IAP 통합
## 🎯 목표
시스템 안정성 지표($R_{Stability}$)를 기반으로 한 구독(IAP) 기능의 백엔드 및 프론트엔드 연동을 통해 프리미엄 가치를 제공한다.

## ⚙️ 핵심 원칙 ($R_{Stability}$-Compliant)
1. **데이터 무결성 우선**: 모든 API 응답은 $R_{Stability}$ 지표의 데이터 파이프라인에서 직접 추출된 값만을 사용해야 하며, 데이터 불일치 시 시스템 오류를 반환한다.
2. **인증 강화**: 모든 IAP 관련 엔드포인트는 반드시 유효한 인증 토큰(JWT 또는 세션 기반)을 요구하며, HTTP 401/403 에러 처리 로직을 명확히 정의한다.
3. **안정성 우선 UX**: 시스템 오류 발생 시, 결제 유도 대신 안정성 확보 방법을 안내하는 디자인 및 메시지 흐름을 백엔드에서 지원해야 한다.

| Endpoint | Method | Description | Required Auth | Stability Check | Notes |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `/api/v1/stability_data` | GET | 시스템 안정성 지표($R_{Stability}$) 조회 | Bearer Token | 필수 | 데이터 파이프라인 직접 연동 |
| `/api/v1/iap/purchase` | POST | IAP 상품 구매 요청 (결제 처리) | Bearer Token | 필수 | PayPal 통합 로직 포함 |
| `/api/v1/user/status` | GET | 사용자 안정성 상태 조회 | Bearer Token | 선택 | 개인화된 신뢰 지표 제공 |

## 🔧 구현 가이드라인
1. **백엔드 (API):**
    *   $R_{Stability}$ 데이터 추출 로직은 파일 시스템 접근 권한 및 경로 오류를 완전히 회피하도록 재검토한다. (이전 단계에서 해결되었으나, 안정성 확보를 위해 최종 확인 필요).
    *   결제(IAP) 로직은 외부 서비스(PayPal)와의 통신 중 발생할 수 있는 모든 예외 상황을 처리하는 강력한 가드(Guard)를 구현한다.
2. **프론트엔드 (UI/UX):**
    *   모든 CTA(Call to Action) 배치는 API 응답의 HTTP 상태 코드($R_{Stability}$ 연동)에 따라 동적으로 변경되어야 한다.
    *   시스템 불안정 시, 사용자에게 좌절감을 주지 않고 안정성 확보 방법을 안내하는 메시지 템플릿을 준비한다.
